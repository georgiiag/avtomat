# Интеграция Avtomat → SELMA HUB (Production)

Цель этого документа — дать разработчику SELMA HUB и ИИ‑агенту ясную “карту переноса”: какие функции уже есть в Avtomat, чего нет, какие риски, и как аккуратно встроить это в корпоративную систему (БД, auth, observability, роли, хранение файлов).

## 1) Что Avtomat делает хорошо (и что можно переиспользовать)
- UI‑конструктор ТП: моделирование операций/переходов и работа со справочниками внутри одного экрана.
- Парсинг спецификаций:
  - DOCX/DOC → BOM JSON (`POST /api/parse_bom_docx`)
  - Excel → BOM JSON (`POST /api/parse_bom_excel`)
- Экспорт BOM и сохранение HTML‑снапшота (как “отчёта”).

## 2) Что “не корпоративно” в текущей реализации (gap‑list)
- Нет БД → состояние хранится в localStorage (пер‑пользователь/пер‑устройство).
- Серверное файловое хранение идёт в `app/static/...` → на Railway это ephemeral и небезопасно.
- Нет аутентификации/авторизации, CORS открыт всем.
- Монолитный фронтенд в одном HTML‑файле без модульности/сборки.
- Нет разделения “domain API” и “presentation”.

## 3) Предлагаемый вариант интеграции (рекомендуемый)

### 3.1 Сервисная граница
Разделить на 2 логических домена:

1) **TP/BOM Service** (часть SELMA HUB backend)
- Хранит сущности: Product, BomVersion, BomItem, Operation, Transition, Registry, Attachment.
- Даёт API для UI SELMA (CRUD и поиск).

2) **Spec Recognizer Service** (может быть отдельным микросервисом)
- Чистая функция: файл спецификации → нормализованный BOM.
- На старте можно временно оставить реализацию Avtomat как отдельный сервис (Flask) и интегрировать в SELMA через HTTP.

Почему это удобно:
- распознавание обычно “тяжёлое” и имеет отдельные зависимости (LibreOffice/Word automation/openpyxl)
- можно масштабировать отдельно
- можно изолировать безопасность (upload only, rate limit)

### 3.2 UI стратегия
Есть 2 пути:

**Вариант A (рекомендуемый): переписать UI под SELMA Frontend**
- Avtomat становится источником доменной модели и UX‑референсом.
- SELMA UI использует “корпоративные” компоненты, auth, роутинг, state management, доступ к БД.

**Вариант B (временный мост): встраивание текущего tp_constructor как iframe/страницы**
- Быстро даёт команде привычный инструмент.
- Но придётся решать auth и безопасное хранение данных/файлов, а также интеграцию с БД через новые API.

## 4) Маппинг доменных сущностей Avtomat → SELMA
См. детали: [DOMAIN_MODEL.md](DOMAIN_MODEL.md)

Минимальный маппинг:
- Product → `production.products`
- BOM version → `production.bom_versions`
- BOM item → `production.bom_items` (tree с parent_id / materialized path / nested set)
- Operation → `production.operations`
- Transition → `production.operation_transitions`
- Attachments → `files` (S3‑совместимое + метаданные в БД)
- Registries → справочники SELMA (обычно отдельные таблицы)

## 5) Хранение файлов (обязательная замена для прод)
As-is:
- `/api/upload_attachment` и `/api/save_tp_html` пишут в `app/static/...` (эпhemeral).

To-be:
- хранение в object storage (S3/MinIO) + CDN
- в БД хранить метаданные (owner, product_id, bom_item_id/op_id, type, mime, size, checksum, created_at)
- ввести ограничения: max size, allowlist MIME, антивирус/сканер (опционально)

## 6) AuthN/AuthZ (обязательная замена для прод)
As-is: нет auth.

To-be:
- SELMA SSO / корпоративная учётка
- роли минимум:
  - технолог (редактирует ТП/справочники)
  - просмотр (читает, экспорт)
  - админ справочников
- аудит: кто/когда менял операцию, кто загрузил спецификацию

## 7) Контракты интеграции (что можно использовать сразу)
- Прямое подключение к распознаванию:
  - `POST /api/parse_bom_docx`
  - `POST /api/parse_bom_excel`
  - `GET /api/status`

Рекомендация:
- завернуть распознавание в SELMA endpoint:
  - `POST /production/spec/parse` → внутри вызвать recognizer service
- затем сохранять результат уже в БД SELMA (а не в localStorage).

## 8) План миграции (этапы)
1) Зафиксировать “контракт BOM” (формат JSON на выходе распознавания) и сделать его стабильным.
2) В SELMA создать таблицы и CRUD для Product/BOM/TP.
3) Вынести распознавание как сервис (или встроить код распознавания в SELMA backend как модуль).
4) Реализовать UI SELMA для конструктора ТП (минимальный функционал: импорт BOM + список операций).
5) Подключить справочники (центры/оснастка/материалы/производители).
6) Подключить файлы через object storage.
7) Убрать localStorage как источник правды; добавить миграцию/импорт из localStorage (опционально).

## 9) Риски и “острые углы”
- “Залипание” старых данных в браузере (localStorage) — при переносе в SELMA нужно явно обозначать пользователю “данные хранятся в системе, а не в браузере”.
- Распознавание офисных файлов в прод окружении:
  - `.doc` требует LibreOffice/Word automation → лучше стандартизировать вход на `.docx`
  - Excel‑парсинг зависит от `openpyxl` (в окружении может отсутствовать)
- Безопасность загрузки файлов и CORS.
